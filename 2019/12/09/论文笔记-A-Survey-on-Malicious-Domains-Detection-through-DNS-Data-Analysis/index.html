论文笔记:A Survey on Malicious Domains Detection through DNS Data Analysis
发表于 2019-12-09| 更新于 2019-12-25|网络安全

文章地址在这里。

ABSTRACT

在这篇论文中，作者介绍了一些关于DNS和利用DNS进行恶意活动的背景知识。描述了一个使用DNS数据的恶意域检测技术的通用框架。应用该框架，我们使用多个正交的观点对现有的方法进行分类，即（1）DNS数据的来源及其丰富，（2）数据分析方法，和（3）评估策略和度量。在每一个方面，我们都讨论了研究社区应该解决的重要挑战，以便充分认识到DNS数据分析在对抗利用恶意域的攻击方面的能力。

1.INTRODUCTION

Domain Name System(DNS)域名系统协议是Internet的基本组成部分。它将难以记忆的IP地址映射到容易记忆的域名。通过分析DNS数据检测恶意域名有很多优点：

  • DNS数据只占整个网络流量的一小部分，使得它在即使是覆盖很广的大规模网络中也适合分析。
  • 缓存(caching)，作为协议中不可缺少的一部分，自然得有利于减少要分析的数据量，这使得研究人员可以甚至分析达到顶级域名(Top Level Domains)的DNS流量。
  • DNS流量中包含大量有意义的特性来帮助识别与恶意活动相关联的域名。
  • 这些特性中的许多可以进一步丰富相关信息，例如AS number，域所有人。从而提供更丰富的用于检测的空间。大量的特征和数据使得其很适合许多机器学习算法应用于安全上下文。
  • 尽管加密DNS数据例如DNSCrypt存在，仍然有很大一部分DNS流量仍然未加密，这使得它可用于多种互联网优势点(vantage point)检查。
  • 研究人员能够在攻击的早期，甚至是攻击之前，从DNS数据中留下的一些痕迹揭示攻击。

本文的目的是通过对DNS数据的分析，综述各种检测恶意活动域的方法。作者从多个来源收集文献，建立了一个全面的书目。

本文提出了一个通用框架（如图1所示）来描述实现基于DNS的检测技术所需的各种组件。它包括以下关键组成部分。

  • DNS数据收集(Data Collection)DNS数据在DNs体系结构的不同位置以不同粒度收集。可能以详细的DNS查询/响应日志(DNS query/response logs)提供，或者仅仅是聚合的形式。
  • DNS数据扩充(Data Enrichment)通常的数据扩充需要集成来自不同来源的网络和应用程序数据。比如域注册记录(domain registration records)、自制系统号(autonomous system numbers)和IP托管域的位置信息(geo-location information of IPs hosting domains)。
  • 算法设计(Algorithm Design)
  • 正确标注的数据(Ground Truth)在监督学习和半监督学习中，包括训练和评价阶段都需要Ground Truth。
  • 评估方法(Evaluation Methodology)除了一些机器学习的评估方法，因为恶意域名检测有高度动态性和攻击者的适应性。所以还需要评估对自适应攻击者的算法鲁棒性，即攻击者可能会估计改变行为还躲避检测。

2.DOMAIN NAME SYSTEM BACKGROUND

2.1 Domain Name System Operation

域名系统是一个分层的分散命名系统(a hierarchical decentralized naming system)，将服务的物理地址(即IP地址)与其逻辑地址（即域名）分离，以便人们可以仅使用其域名连接到服务。
域名被组织为后缀树结构(suffix tree)，称为域命名空间(domain namespace)。最右边的标签命名为顶级域名(Top Level Domain,TLD)。TLD左侧的域是二级域名(Second Level Domain,2LD)。全限定域名(Fully qualified domain name,FQDN），同时带有主机名和域名的名称。资源记录(Resource Records,RRs)，由所有者owner（指示拥有资源记录的DNS域名）、类型type（指定RR类型的编码16位值）、生存时间time-to-live,TTL值（指明其它DNS服务器在期满放弃该记录信息之前对其缓存多长时间）和RDATA字段定义，RDATA字段的内容和语义取决于类型字段的值。本文对以下类型的RR感兴趣：A/AAAA存储IPv4/IPv6主机地址；NS指向存储域信息的权威服务器；MX用于确定邮件的发送位置；PTR记录将主机的IP地址映射到其域名或主机名。

2.2 Domain Name System Security

DNS安全可以划分为四个子区域：

  • 保护DNS(Securing DNS)
  • 保护DNS提供的数据(Securing Data Provided by DNS)
  • 保护用户免受利用DNS的恶意攻击(Securing Users from Attacks Leveraging DNS Disingenuously)
  • 真正保护用户免受利用DNS的攻击(Securing Users from Attacks Leveraging DNS Genuinely)

DATA SOURCES DEFINITIONS

3.1 Source of DNS DATA

DNS数据收集：

  • 在哪里收集数据。在图一中的resolver(the resolver is unique as it is the only location which has access to queries coming directly from client machines)是唯一的可以直接访问来自客户机的查询。作者将数据源分为两种：
    • Host-Resolver。观察终端主机和其解析器之间的通信。图一中的连线1和8。这种来自resolover内部接口的数据一个主要的好处就是提供了客户端关于DNS请求和响应的详细信息。这可以帮助直接将其和恶意行为的类型联系起来。但是它也有一个限制，就是它只能看到一个单一组织内部的主机行为，所以可能不够全面。并且由于隐私原因，公共DNS服务器或者ISP DNS服务器不会轻易将其DNS数据日志提供给研究机构。
    • DNS-DNS。观察DNS服务器之间的通信。图一中的连线2和7。最常见的观察DNS-DNS流量的地点有两个，一是在权威域名服务器(authoritative name server)，包括负责顶级域名服务器(TLDs)；二是在解析器的外部接口。检测器越接近DNS树的根节点，他所能收集到数据就更广。但是，研究人员很难从域名服务器获得日志。
  • 如何收集数据。
    • 主动DNS数据收集(Active DNS Data Collection)。数据收集器会主动发送DNS查询并记录相应的DNS响应。查询域的列表是由多个源创建的，典型的源包括流行域列表，如Alexa Top站点[25]、出现在各种黑名单中的域或来自权威服务器区域的域。显然，由于查询是由数据采集器发出的，因此它们不能反映实际用户的行为。相反，主动DNS数据主要捕获域的DNS记录，例如，解析的IPS、规范名称、记录的TTL等。这种方法的主要优点是灵活且数据易用。此外，主动DNS可以在域被恶意使用之前显示有关域的滥用信号。例如，主动DNS收集器可以发现一个新注册但尚未使用的潜在恶意域，而被动传感器还看不到它。此外，主动DNS数据与个人用户的行为没有联系，因此，可以与研究社区共享，而不必担心隐私问题。同时因为这个原因，主动DNS数据不能用于检测基于用户级特征的技术（例如，用户查询的时间统计）的恶意域。如果DNS查询仅从一组有限的主机发出，则收集的数据可能会有偏差，这是这种方法的另一个限制。具体来说，如果查询者的地理位置不同，查询的域可能关联到不同的ip。
    • 被动DNS数据收集(Passive DNS Data Collection)。被动地观察发送到DNS服务器的所有请求，提取必要的数据。被动收集DNS数据是通过在DNS服务器前部署传感器(sensor)或通过访问DNS服务器日志来获得真实的DNS查询和响应。因此，被动收集的DNS数据在一组丰富的特征和统计数据方面更具代表性和“揭示性”，这些特征和统计数据可用于识别恶意活动。此外，如果sensor部署在来自不同地点的不同组织的DNS服务器上，被动收集的DNS数据可能比主动收集的数据更全面。另一方面，由于隐私问题，此类数据的共享可能会受到阻碍，特别是在客户端和解析器之间部署传感器的情况下。因此，现有的公开可用的被动DNS数据集是在解析器之后收集的，并且通常只提供聚集的查询视图来隐藏单个活动。此外，对于给定的域和它的一个解析IP，它只记录第一次和最后一次看到的解析的时间戳以及它们之间的总数。这是隐私保护、共享能力和数据效用之间的权衡。与主动获取的DNS数据类似，不可能从该数据集构建ne-grained用户级特性。然而，由于聚合是在查询和响应上进行的，因为实际的主机/用户活动，仍然可以导出一些重要的聚合用户统计信息，这些统计信息对于恶意域检测可能非常有用。例如，即使在聚合之后，仍然可以在短时间内观察到一组域上的查询突然增加。从主动收集的DNS数据中无法获得此类统计数据。

挑战：
研究界在获取DNS数据方面面临的挑战有两个方面。
第一个是在数据收集阶段。虽然DNS流量存在于所有网络中，但收集数据集并不是一件容易的事情。如前所述，在单个组织的网络（如校园网）中设置DNS流量传感器相对容易，但随后收集的数据只能提供有限的全局威胁的本地视图。许多现有的基于DNS的恶意域检测技术的特点是它们在大数据场景中工作得最好。因此，它们可能无法在小型网络中收集的数据集上产生有意义的结果。同时，由于DNS日志的敏感性，集成来自不同组织的DNS服务器的数据常常会面临重大的法律障碍。如果研究人员希望从公共DNS服务器或ISP获取数据，情况也是如此。
更大的挑战在于数据共享。与安全相关的数据是众所周知的敏感和难以共享。即使研究人员能够从ISP访问DNS日志，也很难将相同的数据提供给对等方进行验证。同时，科学进步依赖于对现有方法的验证和比较。有人试图将新方法与以前的方法进行比较，但是目前还没有一组公开通用或可比较的参考数据集。
此外，研究人员必须确保使用特定数据集获得的结果可以推广到所有其他可能的数据集。显然，有些数据集可能具有空间或时间特性，这会对结果产生很大影响。

3.2 Sources of Data Enrichment

为了深入了解恶意活动并提高准确性和覆盖率，本篇文章中提出的大多数检测方案利用外部数据源来丰富DNS信息。通常，数据丰富的来源可以根据它们提供的信息类型进行分类。
Enrichment Information Types:

  • 地理位置(Geo-location)。IP和域名的地理位置通常可以用于理解DNS查询的源头。有关地理位置的信息通常可以在Maxmind数据库找到。
  • 自治系统号(Autonomous system number,ASN)。使能了解敌方资源的分布和利用情况。比如说合法的域(除了使用CDN的域)通常托管在一个或几个ASN上，但是恶意域会经常更换ASN来逃避检测。有关IP-ASN映射的信息可以在Maxmind数据库中找到，也可以使用Cymru服务。
  • 注册记录(Registration records)。注册记录的时间信息(比如创建和到期时间)很重要，一些域会自动大量注册以用于后续的恶意行为。注册记录信息通常是从通过WHOIS歇息提供访问权限的服务器获得的。
  • IP/域 黑名单/白名单。比如通过检查和一个域相关的IP中有多少被列入黑名单，准备认为是该域恶意的一个指标。还比如，如果某个域的权威域名服务器被列入黑名单，则认为该域可疑。
  • 关联资源记录(Associated resource records)。探索DNS数据库检索到其他与其相关的RR可以获得有关给定域或IP的更多信息。
  • 网络数据(Network data)。网络活动可以用来丰富信息。例如，如果一个网站和域名相关联，它的HTTP响应是什么，哪一个端口被打开等等。研究人员通常通过开发自己的探针(probes)或者使用网络扫描仪比如Censys和Shodan来获取此类信息。

挑战
和IP或者域相关联的信息是随时间的变化而变化的。由于研究人员经常使用历史DNS数据，因此他们必须依赖于和DNS数据收集同一时间的信息。第二，由于IP地址众多，域名数量快速增长，相应的丰富数据频繁变化，相关历史信息的维护和管理需要大量的资源，研究人员可能无法利用这些资源。

3.3 Source of Ground Truth

通常来说每种用于检测恶意域名的方法都依赖高质量的真实数据，用于训练和验证。
真实数据的类型：

  • 真实的恶意域名(Malicious Ground Truth)。现有的方法主要是从各种公开的黑名单中提取。
  • 真实的良性域名(Benign Ground Truth)。文献中的benign ground truth通常来自排名很高的流行域名。比如来自gov和mil的域名或者属于谷歌和微软的域名被认为比来自com或info的域名可信。

挑战

  • 真实的恶意域名面临的挑战(Malicious Ground Truth Challenges)。首先，恶意域名可能以不同的方式进行恶意行为：垃圾邮件、网络钓鱼、C&C、不道德、成人内容等。因此，不同数据集对真实数据的定义不同。其次，黑名单采用不同的收集方法。第三，没有一个黑名单是完全可靠的。
  • 真实的良性域名面临的挑战(Benign Ground Truth Challenges)。，恶意域可能不会被列入黑名单，因为它在扫描时没有暴露恶意内容，或者从未被扫描过。
  • 普遍挑战(Common Challenges)。对子域的恶意或者良性难以判断。此外，训练集数据不平衡也会有很大的影响。

4 DESIGN OF DETECTION ALGORITHMS

4.1 Features

特征提取需要考虑到准确性和鲁棒性的平衡。很少有方法可以直接利用DNS记录中的字段，大部分都要对这些原始值进行多种处理，例如平均等。此外，某些方法还需要将DNS数据转换为其他数据格式，例如图。

三个维度：
1.内部和上下文特征(Internal vs. Contextual features)
内部特征：这些特征可以直接从DNS资源记录(RRs)中提取，无需外部数据源，但大多数情况下可能都会需要进行转换。
上下文特征：上下文特征是从DNS记录和外部信息源的组合得到的。尽管上下文特征存在一些问题比如隐私问题，成本过高等，但是它能提供更广泛的恶意行为的信息。
2.DNS数据依赖和数据独立特征(DNS dataset Dependent vs. Independent features)
单独依赖数据依赖特征的方法会很容易受到选择的数据集的影响，所以评价这类方法需要交叉验证集测试。相反，数据独立特征在不同环境下更稳定。
3.单一域名特征和多域名特征(Mono vs. Multi Domains Features)
单一域名特征是从每个单独域名中提取的。
多域名特征是从一对域名中得来的，常常被用来进行图分析和聚类。

4.2 Detection Methods

1.基于知识的方法(Knowledge Based Approaches)
基于知识的方法非常依靠专家的洞察力，而且通常专家不能挖掘出高维度的特征关系。
2.基于机器学习的方法(Machine Learning Based Approaches)
大多数的恶意域名的检测方法都是基于机器学习算法的。现有的机器学习方法可以按照数据集标注的不同分为：无监督学习，半监督学习和监督学习。
3.混合方法(Hybrid Approaches)
现有的大多数方法都是混合方法，它可以是多种机器学习算法的组合。也可以是机器学习和基于知识的方法的组合。

4.3 Outcome

在提出的许多方法中，有些方法能够识别特定类型的“恶意行为”，而另一些方法则无法解释为什么它们判定某个域是恶意的。因此，根据结果将方法分为检测特定恶意行为的方法(Malicious behavior specific approaches)和检测不定恶意行为的方法(Malicious behavior agnostic approaches)。
